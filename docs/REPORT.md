# 慶應生向け教科書・授業ノート売買アプリ 制作レポート

2026年「実践のためのWebプログラミング」のレポート課題です。

## テーマ

慶應義塾大学の学生同士で、授業に必要な教科書や授業ノートを探し、学内で受け渡しできるWebアプリを制作した。

## サービスの説明

一般的なフリマアプリでは、大学の授業名や学部・学科から教材を探しにくい。そこで、利用者を慶應生に絞り、授業と教材を関連付けて検索できるサービスを考えた。学内での手渡しを前提にすることで送料をなくし、購入者は安く入手でき、出品者は不要になった教材を次の学生へ渡しやすくすることを目指した。

主な機能は以下のとおりである。

- 慶應メールアドレスとパスワードによるアカウント登録・ログイン
- 学部、学科、学年、授業名、価格、GPAによる商品の検索・並び替え
- 1〜10枚の写真を付けた教科書・授業ノートの出品
- 商品詳細、出品者プロフィール、受け渡し希望の表示
- いいねと「いいねした商品」一覧
- 出品者と購入希望者のリアルタイムチャット、画像送信、未読通知
- 出品者本人による出品の取り消し
- ログアウト、一定期間後のセッション終了、アカウント削除
- スマートフォン向けメニューとレスポンシブ表示
- 写真ライブラリやカメラから選べるプロフィール画像設定

## 設計の説明

### 全体構成

アプリはフロントエンド、バックエンド、データベースの三層構成にした。

```text
React + TypeScript
  ↓ JSON API / Socket.IO
Node.js + Express
  ↓ Prisma
PostgreSQL
```

フロントエンドはReact、TypeScript、Viteで構築した。検索条件やモーダル、商品詳細、チャットなどの画面状態をReactで管理し、Express APIから取得したデータを表示する。スマートフォンでは操作ボタンが潰れないよう、アカウント、チャット、出品、ログアウトをハンバーガーメニューにまとめた。また、リロード後も商品詳細やチャット画面を復元できるよう、現在の画面をブラウザのセッションに保存した。

バックエンドはNode.jsとExpressを使用した。商品検索、出品、いいね、チャット、認証、アカウント削除などをAPIとして実装した。チャットはSocket.IOを利用し、定期取得ではなくリアルタイムでメッセージを反映する。APIでは、ログインしている本人をサーバー側で判定し、他人の出品やアカウントを削除できないようにした。

データベースにはPostgreSQLを使用し、Prismaで操作した。主なテーブルはUser、Course、Item、ItemImage、Like、Demand、ChatRoom、Message、Transaction、Review、LoginAttemptである。商品に出品者情報を重複保存せず、ItemとUserをリレーションで結び付けることで、出品者の学部、学年、GPAを検索に利用できるようにした。

商品画像、チャット画像、プロフィール画像はデータベースへ直接保存せず、CloudinaryへアップロードしたURLを保存する。これにより、Renderのサーバーが再起動しても画像が消えない構成にした。プロフィール画像のURLはUserに保存し、ヘッダー、マイアカウント、商品詳細、出品者プロフィールへ共通して反映する。画像が未設定の場合は利用者名の頭文字を表示する。本番環境は、ReactをRenderのStatic Site、ExpressをWeb Service、PostgreSQLをRender Postgresへデプロイした。

### セキュリティ面

JWTを基本的にHTTP-only Cookieへ保存し、JavaScriptから直接読み取れないようにした。ただし、RenderではフロントエンドとAPIが異なるURLになるため、SafariなどでCookieが制限され、何度もログインを要求される問題が発生した。そこで、Cookieを利用できないブラウザでは、有効期限付きの認証トークンをAuthorizationヘッダーとSocket.IOの接続情報で送るフォールバックを追加した。新規登録が成功すると認証情報を発行し、改めてログインフォームを入力せず、そのままログイン済みの状態で利用を開始できる。ログインに5回失敗すると15分間ロックし、入力ミスへの余裕を持たせながら、総当たりによるログイン試行を抑える仕組みも追加した。さらに、CORSによる接続元の制限、HelmetによるHTTPヘッダー設定、APIのレート制限、慶應メールドメインの確認を実装した。

## 自分なりにこだわった点

最もこだわったのは、単に商品を一覧表示するだけでなく、実際に学生が使う場面を考えて入力や操作を改善したことである。

最初は出品画像をURLで入力する設計だったが、スマートフォンで画像URLを準備して貼る操作は使いづらいと感じた。そこで、iPhoneの写真ライブラリやカメラから画像を選択できる形式に変更し、最低1枚、最大10枚まで添付できるようにした。チャットでも写真を送れるようにし、教科書の書き込みや傷の状態、受け渡し場所などを画像で確認できるようにした。

学部と学科も自由入力では表記揺れが起こるため、選択式にした。先に学部を選び、その学部に存在する学科だけを次の選択肢に表示するようにした。例えば法学部を選ぶと法律学科と政治学科が表示される。このように、データの正確さと入力しやすさを両立させることを意識した。

受け渡し場所については、出品時点で完全に決定させるのではなく、出品者が大まかな希望を入力し、購入希望者とチャットで詳細を決める流れにした。実際の学生同士の予定は変わりやすいため、最初から細かく固定するより自然だと考えた。

デザインは慶應らしい紺色を中心にしながら、フリマアプリとして堅くなりすぎない色や余白を意識した。PC画面だけでなくスマートフォンでも確認し、ボタンが横につぶれた問題をきっかけにメニューバーを追加した。見た目だけではなく、必要な操作へ確実に到達できることを重視した。

## 学びになったと感じたこと

今回特に学びになったのは、ローカルで動くことと、インターネット上へデプロイして動くことは別の問題だという点である。最初はプロジェクトのルートにpackage.jsonがなく、実行する場所によってnpmコマンドが失敗した。またRenderでは、Root Directory、Build Command、Start Commandの違いが原因で何度もデプロイに失敗した。

Prismaについても、マイグレーションが成功していてもPrisma Clientが生成されていなければサーバーは起動できないことを知った。さらに、ローカルでは使えたtsxがRenderではdevDependenciesから除外されて起動できない問題もあった。エラーログを読み、「データベース」「ビルド」「実行時の依存関係」のどこで失敗しているのかを分けて考えることが重要だと分かった。

認証では、画面上でログイン中に見えていても、サーバー側のCookieが期限切れなら保護されたAPIは実行できない。フロントエンドの表示状態とバックエンドの認証状態を一致させる必要があることを、アカウント削除時のエラーから学んだ。また、パスワード、データベースURL、CloudinaryのAPI SecretなどをGitHubへ公開せず、ローカルでは.env、RenderではEnvironment Variablesへ保存する理由も理解できた。

最初にすべてを完璧に設計するのではなく、実際に触って「画像URL入力は使いづらい」「スマホではボタンが潰れる」「リロードすると元の画面へ戻る」といった問題を見つけ、少しずつ改善することもWeb開発の重要なプロセスだと感じた。

## 感想

当初は、学部やGPAで商品を検索するAPIを作ることから始めたが、開発を進めるうちに、出品、画像、プロフィール、ログイン、いいね、チャット、通知、スマートフォン対応まで必要な機能が増えていった。機能同士がつながると、一つの変更が別の画面やデータベースにも影響するため、想像していたより難しかった。

一方で、エラーを一つずつ解決し、最終的に自分のスマートフォンから公開URLへアクセスできたときは、ローカルで画面が表示されたときとは違う達成感があった。特に、自分が「使いづらい」と感じた部分をそのままにせず、写真添付や学部・学科の連動選択、スマートフォン用メニューへ変えられたことが良かった。

今後は、慶應メールの確認メール送信、取引状態の更新、取引完了後の相互評価、違反商品の通報や管理者機能などを追加したい。また、授業ノートや過去資料を扱う際の著作権・大学のルールも確認し、安全に利用できるサービスへ改善する必要がある。
